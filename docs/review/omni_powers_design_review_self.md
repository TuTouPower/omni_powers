# omni_powers_design.md 审阅报告（主会话）

## 审阅范围

- 审阅对象：`docs/omni_powers_design.md`
- 审阅日期：2026-07-07
- 审阅人：主会话
- 审阅维度：结构完整性、需求/状态机一致性、heavy/lite 分叉一致性、Agent 协作边界、可执行性、风险与遗漏
- 约束：只生成审阅报告，不修改原设计文档

## 总体结论

文档整体成熟。它不是简单流程说明，而是把 omni_powers 的核心取舍、阶段目标、当前限制和未来安全模型放在同一份设计档案里解释。最值得肯定的是：没有把尚未落地的 worktree 隔离、CI 证据链、evaluator 独立验收包装成当前能力，而是在 §0.1、§8.1、§10、§12 多处明确区分当前 advisory 与目标硬隔离。

结构层面，文档按“原则 → 流程 → spec/task → 执行循环 → 验收 → 护栏 → 插件结构 → 路线 → heavy/lite 分叉”展开，顺序合理。heavy/lite 的差异被压缩到 §13 的环境集成层，避免两套流程分裂。lite 的退化也在 §14 明确列出，整体诚实。

主要问题集中在三类：目标态与当前态反复交织导致阅读负担；部分关键开放问题已登记但未给执行前置条件；lite 的上下文风险和迁移路径还不够硬。

## 主要问题

### CRITICAL

无。

### HIGH

#### H1. 当前态/目标态分散标注，读者容易误判能力边界

§0.1 写明 P2 前 heavy 防篡改增量接近零；§2、§8.1、§10、§12 又多次说明 worktree 隔离、CI 证据链、e2e 只读回传属于目标态。每处都诚实，但信息分散。读者若只读 §2 或 §10 局部，容易误以为结构隔离已落地。

建议在 §0.1 后增加“当前能力快照表”：防线、当前状态、目标状态、落地阶段、失效后果。后文只引用表，减少重复声明。

#### H2. e2e 合法写入通道是 P2 前置设计，不宜只作为 P2 同步事项

§8.3 已登记问题，但 P2 同时要落 worktree 隔离、git 硬锁、CI 证据链、evaluator 构建产物链路。若 e2e 写入通道与 git 锁一起才设计，容易影响整条 P2 主线。

建议把 §8.3 拆成 P2 前置 ADR：先确定“谁能改 e2e、通过什么入口、如何审计、如何失败回滚”，再实现硬锁。

### MEDIUM

#### M1. spec 变更子流程定义偏短

§5.2 用“agent 提 delta → 人批 → 重新 commit → 受影响 task 失效重拆”概括变更流程。对原则说明够用，但作为运行设计，缺少 delta 文件位置、发起者、审批归属、task 失效范围判定、重拆后旧 task 状态。

#### M2. lite → heavy 迁移语义未完整定义

§5.1 保留 `feature` 字段便于 lite→heavy 迁移，§14.3 也提到未来迁移。但 lite 无 blueprint 真相源、无 baseline 对照、无 hook/worktree 结构。迁移不是切 profile，而是补建真相源与重验收。

建议在 §13 或 §14.3 明确：lite→heavy 是显式迁移流程，不是自动切换；至少需要 `/opinit` 类初始化、blueprint 补建、已有 spec 归档关系校验、关键 E2E 重跑。

#### M3. lite 上下文水位保护仍偏人工纪律

§14.1 识别了 lite 的根本张力：leader 自验导致上下文膨胀。当前表述是每 N task 提示 `/compact` 或建议转 heavy。对于“静默失能”这类风险，仅提示不够。

建议将水位检查写成流程门：超过阈值暂停；用户选择 compact、拆分、继续承担风险，且记录 decisions。

#### M4. decisions.md 多来源 append 缺机械幂等说明

§7.4 说明 `decisions.md` 是多写者 append-only 文件，并通过 task 严格串行规避冲突。正常路径成立，但恢复、重试、子 agent 超时后回报等异常路径会产生重复追加或顺序不明。

建议每个 append block 带唯一标识：`source/task_id/round/hash`，post 脚本检查重复。

### LOW

#### L1. §10 正文与防线映射表重复

§10 的正文与表格都承载“当前/目标/是否 hook”信息。建议状态信息只保留在表格，正文只解释机制。

#### L2. §11 脚本清单对 lite 差异提示不足

§11 说 lite 自带副本见 §13.4，但 `op_close_pre.sh + op_close_post.sh` 行容易被读成两版都有。建议在该行标注 lite 砍 `op_close_pre.sh`。

#### L3. §15 信息密度过高

§15 同时写落地状态、新增决策、用户裁决、归档指引。建议保留当前落地状态，其余决策史移动到 `docs/op_decisions.md` 或只保留回链。

## 具体建议

1. **新增当前能力快照表。** 放在 §0.1 后，列出 behavior layer、evaluator 隔离、implementer e2e 隔离、CI 证据链、spec 写保护、lite P0 阻断等当前/目标状态。
2. **将 §8.3 升级为 P2 前置设计项。** 明确 e2e 合法写入入口、审计字段、回滚方式和失败策略。
3. **补全 spec 变更子流程。** 给出 delta 文件路径、审批点、旧 task 标记、重拆规则，避免执行期临场解释。
4. **明确 lite→heavy 迁移不是切换 profile。** 写出迁移代价和最小步骤，避免用户误以为可无缝升级。
5. **lite 水位检查改为流程门。** 连续或严重超阈值时必须暂停，不继续静默推进。
6. **append-only 文件加幂等标识。** `decisions.md`、progress 类文件都可复用同一标记格式。
7. **压缩 §10/§15 重复信息。** 状态归表，决策史归决策文档。

## 值得保留的设计

1. **§0.1 诚实声明。** 明确当前安全水位，防止用户误用 heavy 的目标态承诺。
2. **§5.1 可测性契约。** 把验收信号、通道、测试缝、预期失败模式前置到 spec 期，降低执行期返工。
3. **§5.2 契约边界规则。** 用“是否需要改 spec 文本”划分自决与阻塞，足够机械。
4. **§7.2 review 两轮上限。** 防止模型在错误结构里无限修补。
5. **§8.1 evaluator 防放水机制。** hard-pass gate、预期失败模式、破坏检查、钓鱼审计构成清晰梯度。
6. **§8.2 baseline 三层信号。** 按结构化/视觉/操作分层，比按应用类型枚举更通用。
7. **§13 执行内核与环境集成层分离。** heavy/lite 共享核心，差异集中管理。
8. **§14.3 lite 退化矩阵。** 对每个角色说明退化点，避免模糊承诺。
