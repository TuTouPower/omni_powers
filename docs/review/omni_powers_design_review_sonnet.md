## 总体结论

这是一份**高质量的系统设计文档**，15 节覆盖了从原则到落地路线的全部层面。核心架构决策（两版共存、环境集成层收敛差异、worktree 结构隔离、CI 证据链、双裁决审查）深刻且自洽。D18 带来的"hook 对 subagent 整体失效"认知被诚实贯穿全文档（§0.1 安全增量声明是典范级诚实写作），不再有"配齐 hook = 安全"的错误预期。

主要问题集中在：一个未解决的开放问题可能阻断 P2 安全模型（e2e 写入与 git 硬锁的交互）、lite 的规模上限缺乏自动保护机制、CI 依赖集中度过高形成单点。以下逐项展开。

## 主要问题

### CRITICAL（0 项）

无。现有问题可在 HIGH 级解决，无阻断文档发布的问题。

### HIGH（5 项）

**H1. e2e 合法写入与 git 硬锁的交互未解决（§8.3，P2 设计项）**

文档诚实登记为"开放问题"，但影响面被低估。P2 实施 worktree 隔离后，git pre-commit 硬锁 `e2e/**` 存在三个合法写入路径：evaluator 固化 PASS 测试、leader 转交 BUG-* patch、closer 提案后改跨功能既有 e2e。git hook 层无法区分"谁"在提交（同样面临 D18 式身份识别困境）。若不解决，P2 的硬锁要么把 evaluator 自己也锁死，要么解锁通道形同虚设。文档给出两个候选方向但均未细化，在 P2 实施前必须有明确设计。

建议：leader 主会话作为唯一 e2e 写入入口（pre-commit hook 在 leader 路径放行 + commit trailer 配对），evaluator 产 patch 由 leader 代提交。这与 implementer 产 BUG-* patch 由 leader 转交的模式一致，复用同一信任边界。

**H2. lite leader 上下文膨胀无自动保护（§14.1）**

文档正确指出 lite 的根本张力——leader 每 task 读 report + 跑测试 + 读 diff，N 个 task 后上下文膨胀——并给出了缓解策略。但 "每 N task 自检上下文水位" 是写进契约的，当前 §15 显示该检查"待落 oplrun"。在实现前，lite 用户可能在不知情下跑到第 8 个 task 时"静默失能"。

建议：oplrun 实现 3-task 水位检查时，不应只"提示"，应在连续 2 次阈值告警后自动中断并指引转 heavy。否则用户可能在紧要关头才发现上下文不够。

**H3. P2 CI 依赖集中度过高（§10、§12）**

三个安全关键特性——per-task 证据 CI 化（原则 7）、e2e 只读信号回传（§2）、evaluator 构建产物链路（§8.1 第 1 层）——全部依赖"同一套 CI 基建"。若 CI 搭建延迟或受阻，三个特性同步卡住，P2 安全增量无法交付。文档未讨论 CI 基建的渐进交付策略或最小可行版本。

建议：将三者拆为独立里程碑。优先 per-task 证据 CI 化（最小范围：跑测试、输出结构化结果文件），e2e 只读信号随后，构建产物链路最后（复杂度最高）。三者可共用 CI 配置框架，但独立灰度上线。

**H4. worktree sparse-checkout 的可移植性假设（§8.1、§10、§12）**

整个 P2 硬隔离层建立在 worktree + sparse-checkout 之上。文档未讨论该特性在不同 git 版本 / 平台（WSL、macOS、Windows Git Bash）的可用性和行为差异。Windows 用户在 Git for Windows 下 sparse-checkout 行为可能与 Linux 不同。P0 spike（§12）已规划验证，但 spike 结果会影响 P2 排期和时间线——当前排期未留缓冲。

建议：P0 spike 结论应反馈到 §12 排期表，明确哪些平台可以物理隔离、哪些降级为 advisory。spike 失败不应导致整个 P2 重设计——应保留备选方案（如独立 clone 代替 worktree）在文档中。

**H5. decisions.md 多写者顺序未显式保证（§7.4）**

文档称"多写者不冲突的前提是 task 串行"（§7.4 注脚）。但 per-task 收口有三步：leader 跑 `op_close_pre.sh` → closer append decisions.md → leader 跑 `op_close_post.sh`。closer 的 append 发生在 leader 的两次脚本调用之间，依赖时序正确——若 leader 在 closer 完成前跑 `op_close_post.sh`，decisions.md 可能丢失。此顺序由人为纪律保证，无机制强制。

建议：`op_close_post.sh` 开头检查 decisions.md 最后修改时间晚于 `op_close_pre.sh` 执行时间（或检查 closer 回报信号），否则 die。微小改动，零成本消隐患。

### MEDIUM（6 项）

**M1. BUG-* 回归测试在目录结构中的位置未明确**

§1.1、§1.2、§2 多处提及 `BUG-{id}_*.spec` 回归测试，但 §3 目录结构图和 §3.1 task 工作区布局中均未出现。读者无法从目录结构推断 BUG-* 测试文件放在哪里。

建议：在 §3 目录结构中标注 `e2e/` 下的 `BUG-*` 文件约定，或在 §2 中写明存放路径。

**M2. spec 变更子流程无具体步骤（§5.2）**

文档多处引用"spec 变更子流程"（§5.2、§7.1、§7.3、§10），定义为"agent 提 delta → 人批 → 重新 commit → 受影响 task 失效重拆"。但从未定义谁提 delta、用哪个 agent、输出文件格式、审批走哪个闸门（是闸门 A 的子集还是单独流程）。

建议：若 spec 变更子流程使用频率低，可在 op_decisions.md 中详述；若使用频率高，应在 design.md §5.2 补充 5-8 行流程定义。

**M3. §7.2 升级人裁决"四选一"未定义决策者**

review 两轮到顶后 Critical 残留进入"升级人裁决"，选项为降级接受/重拆/换更强模型/插设计 task。文档未说明谁做这个裁决——是 leader 自动执行某选项、还是呈报用户选择。从上下文看是"人工裁决"。

建议：明确"呈报用户，由用户四选一"，与 §5.2 契约边界规则中"人批"的表述对齐。

**M4. lite->heavy 迁移路径半定义（§5.1、§14.3）**

文档多处暗示 lite 可迁移到 heavy（`feature` frontmatter 字段保留、"便于 lite->heavy 迁移"），但没有定义迁移步骤。lite 无 blueprint、无 hook、evaluator 裸评——迁移意味着补建 blueprint、注册 hook、重新验收。这是一个有代价的操作，文档应该诚实声明，而不是给读者"随时可切"的错觉。

建议：在 §13 差异面表后加一段迁移说明，明确 lite->heavy 相当于重新跑 /opinit（三区重建 + hook 注册 + 已有 spec 需重验收）。

**M5. 定期体检（§10 第 6 层）的实现未定义**

第 6 层"定期体检"被描述为独立 CI 任务，非 Claude Code hook。但任务调度方式、失败如何转 issues、变异测试的工具选择均未说明。这在 P3 范围，但文档应给出最小定义（cron + jq 脚本 + issue 自动生成），否则读者无法判断实现工作量。

建议：在 §10 末尾加一段体检实现概要（一行调度 + 一行脚本入口 + 一行产物路径）。

**M6. §13.4 脚本表中 `op_read_verdict.sh` 的 lite 版差异标记为"无"**

`op_read_verdict.sh` 在 heavy 和 lite 中均存在且内容相同。这意味着两个副本需要同步维护——而 build_lite.sh 只校验不一致，不自动同步。任何对 heavy 版的修改都需手动同步到 lite 目录。虽然 build_lite.sh 会检测漂移，但维护成本是持续的。

建议：若脚本确实无差异，考虑让 lite 版直接通过 `OP_SCRIPT_ROOT` fallback 引用 heavy 版（或至少加注释标注"此文件与 heavy 版相同，修改时须同步"）。

### LOW（4 项）

**L1. §8.1 钓鱼审计收敛标准依赖 P3 基建**

钓鱼审计是"防放水"的第四条防线，但依赖独立验证环境副本（P3）。文档正确标注了依赖关系（§8.1 末尾和 §12 P3 段），但在 §8.1 第 4 层开头读起来像是当前即可运行的机制——实际是 P3 才生效。建议在 §8.1 第 4 层开头加一句"（P3 落地）"以降低读者误读概率。

**L2. §7.5 模型分配表的 token 消耗列是定性判断**

token 消耗排序 implementer > evaluator > reviewer > closer 是合理推断，但缺乏实测数据支撑。尤其 evaluator 用 Opus + computer use，单次绝对 token 可能远超 implementer 的 Sonnet 多轮累计。建议标注"（待实测）"。

**L3. 术语细微不一致**

- §5.1 模板: "验收场景" 但 §8 和 agent 文件普遍使用 "AC"（Acceptance Criteria）。建议在模板注释中指出"验收场景 = AC"。
- RULES.md 使用"下线传播"描述阻塞传播，design.md 未使用此术语。无害但降低全文搜索一致性。

**L4. §15 落地状态段过长**

§15 是文档尾注段，但"新增决策"和"用户裁决"合计约 25 行——这些是 D20 决策记录的重述。建议将决策细节移回 `op_decisions.md`，§15 只留单行引用，避免两处维护漂移。

## 具体建议

### 建议 1: 补充 CI 架构概要节

当前文档将 CI 作为 P2 实施的隐含前提，但从未定义 CI 的范围和最小接口。建议在 §10 或 §12 后加一节（约 15 行）描述 CI 的最小接口：

```markdown
### 10.1 CI 最小接口（P2 基建依赖）
CI 不绑定具体工具（GitHub Actions / GitLab CI / Jenkins 均可），满足三个最小接口即可：
1. 分支 push 触发：跑项目测试套件，输出结构化结果文件（JSON/JUnit XML）
2. 可读跑 e2e：跑 e2e/ 全集，输出结果文件，不写代码
3. 构建产物生产：从指定分支构建可发布的应用程序包，产物归档至可下载路径
```

这样 reader 不需要猜测"CI 基建"到底要做什么。

### 建议 2: 添加快速导航表

830 行的设计文档，没有搜索功能的读者很难定位具体章节。建议在目录前加一个"按问题导航"表：

```markdown
| 你要问 | 去看 |
|---|---|
| heavy/lite 的区别到底是什么 | §13 |
| lite 缺了什么 | §14.3 退化矩阵 |
| hook 为什么不是安全防线 | §0.1、§10（防线层映射表）、op_decisions.md D18 |
| 安全什么时候才真正落地 | §12 P2 完成后 |
| 一个 task 从创建到归档经历了什么 | §7.1 循环 + §7.4 收口 |
| evaluator 怎么保证不放水 | §8.1 防放水四层 |
| spec 里"可测性契约"是什么、为什么重要 | §5.1 模板 ③ |
```

### 建议 3: 原则 7（机器证据）的文本长度偏长

原则 7 正文是一段 15 行的密集叙述，混合了过渡期事实、目标形态、实现路径。建议拆为 3 小点：
- 7a. 当前状态（过渡期，evidence adivisory）
- 7b. 目标形态（P2 CI 化）
- 7c. 不变核心（bash 先算、LLM 再决策）

现存正文的转折词密集（"但 SubagentStop 只能验存在""当前过渡期""诚实结论""目标形态 P2"），读者容易丢失主线。

## 值得保留的设计

以下设计决策在复杂度和表达质量上突出，值得在后续迭代中保护不被稀释：

1. **§0.1 安全增量的诚实声明。** 在安全文档中罕见——明确告诉读者当前的安全水位，并给出作废条件。这是信任基础。P2 落地后切勿删除，应改为"此声明已于 P2 作废，原内容见 archive"。

2. **原则 4 契约边界规则。** "该决策是否需要改 spec 文本" 作为执行期决策是否需人批的唯一判据，机械、可操作、不需要主观判断。这在多 agent 系统中极有价值，因为它消除了"什么时候该停"的灰色地带。

3. **§4 全流程总览的 ASCII 流程图。** 在同一张图中表示了 Stage、闸门、分支（轻量直做 vs 全流程）、回流路径。紧凑但信息密度高。

4. **§13 两版共存架构的"环境集成层"概念。** 将 heavy/lite 差异收敛到 6 个维度，执行内核完全共享——这是教科书级的两版同源设计。尤其 agent 文件一份、靠 fallback 变量区分行为，是低维护成本的关键决策。

5. **§14.3 blueprint 缺失的连锁退化矩阵。** 不是笼统说"lite 功能少"，而是逐消费者列出缺失了什么、退化成了什么。这种"退化矩阵"写法应在其他组件缺失分析中复用（例如 CI 缺失时各防线退化形态）。

6. **§10 防线层与实现手段映射表。** 明确标注哪些防线是 Claude hook、哪些是结构/git/CI。配合 §0.1 的诚实声明，彻底杜绝"配齐 hook = 安全"的误解。这是在 D18 发现后的关键修正。

7. **§8.2 验收基准快照的三层信号模型。** 结构化/视觉/操作三层的分类不按应用类型枚举，而是按信号性质分类——适用于任何有外部可观察产物的系统。这是可复用的验收框架设计。
