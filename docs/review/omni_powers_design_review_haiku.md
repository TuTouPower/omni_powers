# omni_powers_design.md 审阅报告

## 审阅范围

- 文件：`/home/karon/karson_ubuntu/omni_powers/docs/omni_powers_design.md`（830 行，heavy+lite 合并版）
- 交叉验证：`RULES.md` 状态机段、`skills/oprun/scripts/` 脚本清单
- 审阅维度：结构完整性、需求/状态机一致性、heavy/lite 分叉一致性、Agent 协作边界、可执行性、风险与遗漏

## 总体结论

文档质量高。设计自洽性强，heavy/lite 分叉通过「执行内核共享 + 环境集成层差异」单一抽象收敛，逻辑清晰。最大亮点是**诚实声明**贯穿全文（§0.1、§8.1、§10、§14）：明确标注 P2 落地前 heavy≈lite 防篡改水位、hook 对 subagent 失效、各防线当前 advisory 还是硬隔离，没有用「机器证据」之类的话术糊弄。落地路线（§12）分 P0-P3，每段绑实现项，可执行。

主要问题是**目标态与过渡态混合叙述**造成认知负担，以及部分交叉引用指向未实现脚本。CRITICAL/HIGH 级问题无，集中在中低风险项。

## 主要问题

### CRITICAL

无。

### HIGH

**H1. 目标态与过渡态混写，读时易误判当前能力**

§0.1、§2、§8.1、§10、§12 反复出现「目标形态 P2」「当前过渡期 advisory」「P2 落地后此声明作废」。每处都有标注，但散落 8+ 处。读者首次读到 §2「implementer worktree 不挂 e2e/」会以为是已实现的硬隔离，读到 §8.1「实现状态」才知道是 advisory。建议在 §0 顶部加一张**当前能力快照表**（行=防线层，列=当前/目标/Px），一处看全，正文不再重复「当前未落地」。

**H2. 「§10 防线层↔实现手段」表与正文叙述重复且部分冗余**

§10 末尾的映射表（530-541 行）与正文 522-530 行的五层描述信息高度重叠。表本身是好的（一图全览），但正文每层又写一遍「目标 X / 当前 advisory」，读两遍。建议正文五层只写机制描述，状态全收进表。

## 具体建议

1. **加「当前能力快照表」于 §0 顶部**（治 H1）：单表锁定所有「当前/目标/Px」状态，正文防线描述只写机制不写状态。这是最高杠杆改进，能让全文减约 30-40 行重复声明。

2. **§7.4 closer 权限合并单条清单**（治 M1）：per-task 段以「closer 权限：仅写 decisions.md + 转暂存 issue；不碰 git/status/脚本/blueprint」一条收口，避免铁律散落。

3. **§14.0 状态机图补阻塞分支 + 回链 §13.4**（治 M2/M5）：图对称 heavy，文字一段说清「删收口中→砍 op_close_pre→收口瞬时」因果。

4. **交叉引用补章节号**（治 M3/L1/L4）：§6.3「降细拆惩罚」三项、§3 目录树 lite 注释、§1.2 表 style 行，各加 1-2 字章节锚。

5. **§15 新增决策项标「排期见 §12 Px」**（治 L3）：单行省回查。

## 值得保留的设计

1. **§0.1 安全增量诚实声明**：把 heavy 当前相对 lite 的真实增量明算成「流程资产 + 主会话 hook + git 部分」，不夸大安全。同类设计文档罕见。

2. **§13 执行内核/环境集成层二分**：heavy/lite 差异收敛到 6 点表，单点维护 agent 与 spec 逻辑。这是合并版能成立的关键抽象。

3. **§8.1 evaluator 访问隔离三层（结构+脚本+dispatch）+ 钓鱼审计**：把「防放水」从口号落到「植已知 bug 看抓不抓得到」的判别力测试，且诚实标注 P3 基建依赖。

4. **§5.2 契约边界机械判据**（「需要改 spec 文本吗」一句问句分界自决 vs 阻塞）：执行期决策分流的机械规则，避免「灰色地带全停」或「全自决越界」两端。

5. **§7.2 review 两轮上限 + Critical 四选一**：硬上限防烧 token，分流防静默 commit，降级接受强制记 delta 防「规格唯一契约」破裂。

6. **§9 issues 五入口 + P0 默认阻断**：lite §14.2 补偿检查保住闸门 C 缺失的安全语义，单行检查值钱。

7. **§12 P0-P3 排期与 §15 落地状态双视图**：排期是工程视角，落地状态是诚实账，互不替代。
