# omni_powers 对抗性审查与优化建议

> 视角：第一性原理，宏观策略。不纠缠脚本级 bug。
> 范围：**已排除模型强弱问题**（haiku/sonnet 角色可配置，视为正交变量）。
> 审查对象：`RULES.md` + `skills/`（optask/opspec/opplan/opstart/opdebt/opinit）+ 核心脚本。
> 日期：2026-06-29

---

## 核心论断

当前系统优化的是**流程合规**（文件齐全 + AI 声明 PASS），而非**代码正确**。
排除模型因素后，仍有 4 个结构性缺陷需要修，按 ROI 排序：

| 优先级 | 缺陷 | 本质 | 修复 ROI |
|---|---|---|---|
| P0 | 质量门槛无客观锚 | verdict 可自欺，无不可伪造信号 | 极高 |
| P1 | 自治无锚 → 错误复利 | 不停下问 + 验证不可信 = 无声错误指数放大 | 高 |
| P2 | plan 与 coder 智力劳动重复 | plan 写满代码 → coder 沦为转录员 | 中高 |
| P3 | 状态机为不存在的并发过度工程 | 单进程串行场景套分布式复杂度 | 中 |

---

## P0：给 verdict 装不可伪造的地基（最高优先级）

### 问题

全部质量门槛都是主观声明：
- `verdict: PASS/FAIL` 由 reviewer 在文件末行自己写
- `close_check.sh` 只查"文件齐全 / 段落存在 / 盖戳存在"
- `op_close_post.sh` 只查"三份 review 末行是 PASS"

**没有任何一项验证代码真的正确。** 一个把测试删光、三份 review 都写 `verdict: PASS` 的 task，能完美通过收口、归档、commit、标"完成"。整条链路零感知。

根因：用 AI 审 AI。reviewer 读到的"测试通过"也是 coder 在 context.md 的自我声明。链路里没有一个**不可伪造的 ground truth**。

### 改法

引入"客观门 + 主观门"两层，**客观门在前，一票否决**。

1. **新增 `scripts/op_gate.sh <TID>`**：执行项目真实校验，退出码即门槛。
   ```bash
   # 伪代码
   run test  → 非 0 die
   run build → 非 0 die（若项目有 build）
   run lint  → 非 0 die（若项目配置 lint）
   # 校验命令从 docs/omni_powers/op_blueprint/conventions.md 的 [verify] 段读取
   # 无 [verify] 段 → die，强制项目先声明如何验证自己
   ```
2. **`op_close_post.sh` 开头先调 `op_gate.sh`**：客观门不过，任何 reviewer 的 PASS 都无效，直接 die。
3. **每轮 review 派发前先跑 `op_gate.sh`**：客观门不过，连 review 都不必派——直接回 coder 的 fail 轮，省下 3× reviewer 调用。
4. **reviewer 职责降级**：reviewer 不再判断"测试是否通过"（那是客观门的事），只判断客观门盖不住的东西：测试是否**有意义**（非"调用即断言"）、是否覆盖关键路径、架构与 spec 是否一致。

### 收益

- 假测试被真执行戳穿，A 类失败模式消失。
- review 轮次的触发由客观信号驱动，省掉"代码本就错却派 3 个 reviewer 去读"的浪费。
- 这一改同时给 P1 的自治提供了真锚。

### 协议层改动

`RULES.md` 的"review 判定"节加一句：
> **客观门优先**：任何 review/收口前必须先过 `op_gate.sh`（真测试/build/lint 全绿）。客观门不过，verdict 一律视为 FAIL，与 reviewer 声明无关。

`conventions.md` 模板加 `[verify]` 段，强制每个项目声明自己的验证命令。

## P1：自治无锚 → 错误复利

### 问题

`RULES.md`："不停下问用户（除非全部可跑 task 跑完仍剩阻塞）"。

与 P0 叠加：系统自主连跑长链 task，每个都"自认为"通过，直到全做完才停。若 T02 跑偏，T03–T50 全建在错误地基上，用户在终点才见到一堆**自洽的错误产出**。

自治的前提是**可靠的自我纠错信号**。在 P0 修复前，verdict 不可信，自治只会把无声错误用复利放大——这比"频繁打扰用户"严重得多。

### 改法

P0 修好后，自治才有真锚。再叠加两条护栏：

1. **失败聚集即停**：连续 N 个 task（建议 N=2）走到第 3 轮 quality 阻塞 → leader 停下报告，不再硬推。说明地基系统性出问题，继续跑只是放大。
2. **下游依赖客观门联动**：某 task 客观门反复不过被阻塞时，其下游不仅标"跳过"，还应在报告里显式列出"因 {TID} 未通过客观验证而无法验证"的传播链，让用户一眼看到污染范围。

`RULES.md` 的"不做"节，把"不停下问用户"改为：
> 不停下问用户——**除非**全部可跑 task 跑完仍剩阻塞，**或** 短期内 task 连续 quality 阻塞（地基性故障，继续无意义）。

## P2：plan 与 coder 智力劳动重复

### 问题

`opplan` 要求：每 step 含**完整代码块**、禁止 placeholder、禁止"类似 Step N"。

两难：
- plan 真写到"每 step 完整代码" → 写 plan 的人已把活干完，coder 只是把伪代码抄成真代码。**同一份智力劳动做两遍。**
- plan 只写意图不写代码 → 违反"禁止 placeholder"。

无论哪边都浪费。

### 改法

重新切分 plan 与 coder 的职责边界，按"契约 vs 实现"分工，而非"伪代码 vs 真代码"：

- **plan 负责契约**：文件职责、接口签名、数据模型、step 之间的依赖与验收命令、每个 step 要满足的**测试断言**（断言是契约，必须写死）。
- **coder 负责实现**：函数体内部怎么写，由执行者判断。plan 不再要求贴出完整实现代码块。

把 `opplan` 的"禁止 placeholder"重新定义：
> placeholder 指**契约层的含糊**（"添加适当错误处理"""TODO"），不是"未提供实现代码"。plan 必须写死接口、断言、验收命令；实现代码体由 coder 产出。

收益：消除重复劳动；plan 更短、更稳定（契约比实现稳定）；coder 真正承担"实现"职责而非转录。
代价：要求 coder 有判断力——这正是"客观门"（P0）兜底的地方，实现错了测试会红。

> 注：本条与模型强弱正交。无论 coder 用什么模型，"让写 plan 和写代码的人各干一次"都比"干两遍同一件事"经济。

## P3：状态机为不存在的并发过度工程

### 问题

9 状态 + 4 种 blocked_by + 下游传播 + 绕过机制 + git revert 回滚 + 拆表预案 + `op_status.sh` 文件锁。

但 leader 是**单一主会话，task 严格串行**（"代码执行并行……暂不启用"）。文件锁在防不存在的并发。状态越多，半完成态越多：`close_post` 分"归档→写 progress→改状态→git add"四步，中间 crash 即脏态。**复杂度本身成了新故障源，而它防御的故障（并发、乱序）当前架构压根不会发生。**

### 改法

按"当前真实架构=单进程串行"裁剪，遵循 YAGNI：

1. **冻结而非删除并发设施**：文件锁、拆表预案、worktree 串行 merge 等，移到 `docs/op_findings.md` 标注"并发启用时再恢复"，从主协议 `RULES.md` 移除，减少 leader 每次编排要消化的规则量。
2. **收口四步合并为原子脚本**：`op_close_post.sh` 已是单脚本，但要保证**要么全成要么全不成**——失败时回滚已做的部分（如已 `git mv` 要能撤回），避免脏态。或改为"全部操作先暂存、最后一次性提交"。
3. **状态合并评估**：`收口中` 与 `审阅中` 在串行单进程下信息量低（没有别的 agent 会并发读这个中间态）。评估能否砍到 6 态：待规划/待开始/进行中/完成/阻塞/挂起，把"审阅中/收口中"降为 leader 内部步骤而非持久状态，减少 compact 恢复时要重建的态。

> 原则：状态机的复杂度应匹配真实并发度。单进程串行不需要分布式状态协议。

## B（经济模型）的结构性残留 + 三 review 合并 + test 痛点根治

排除模型强弱后，仍有一条结构问题：**一个 task 最坏 11 次 agent 调用**（1 coder + 3 轮 ×3 reviewer + 1 closer）。

三个 review 各有出身（code=ecc，test=superpowers 实战痛点，spec=superpowers），合理性不同。下面分别处理，核心是**最痛的 test review 不靠 review 解决**。

### B.1 为什么 test review 最重却效果最不稳

用户实战痛点：AI 总写不可靠、没用的测试（调用即断言、mock 掉一切、断言写死=实现输出）。

第一性原理：AI 写假测试**不是能力问题，是目标错位**。它的目标函数是"让测试绿"，不是"让测试能抓 bug"。当同一套智能既写实现又写测试，同样的误解同时进入两边，测试天然测不出实现的错。

**用 test reviewer 审，是用另一个 AI 去赌它没有同样的盲区。** 审者与写者同源同盲区——这就是 test review 既重又不稳的根因。再多一个主观 review，期望值不变，方差很大。

### B.2 根治：把"测试是否有效"从主观 review 移到客观门

"测试能否发现问题"是**可机械验证的**，不需要 AI 判断。两个客观信号，并入 P0 的 `op_gate.sh`：

**信号一：哨兵突变（sentinel mutation）——抓假测试的杀手锏**

故意改坏实现，真测试变红，假测试不变。纯机械、零 AI 判断。

```bash
# op_gate.sh 内，对 plan 指定的关键函数：
# 1. 跑测试 → 必须全绿（基线）
# 2. 应用哨兵突变（如 sed 把目标函数的 return 改坏 / + 改 -）
# 3. 重跑测试 → 必须变红（证明测试真能抓到这个突变）
# 4. 还原实现
# 任一关键函数突变后测试仍绿 → die：该函数的测试是假的
```

落地走 80/20，**不上完整 mutation 工具**（mutmut/stryker 太慢太重）：
- `plan.md` 每个关键 step 标注 1–2 个"哨兵突变点"（改哪行、怎么改坏）。
- 收口客观门机械地应用突变、跑测试、断言变红、还原。
- 只覆盖关键函数，不追求全量突变。

**信号二：RED 必须真红且原因正确**

假测试的典型特征——实现还没写时它就能通过（因为它没测真东西）。

- coder 的 TDD RED 阶段（plan 里已有"跑测试确认失败"步骤）必须**留存失败输出**（测试名 + 失败原因）写进 context.md。
- 失败原因必须是"功能未实现"类，不能是语法错/import 错（那是测试本身坏）。
- 这是 coder 流程内自证，强度弱于突变，作为补充。

> 突变信号强于 RED 信号：突变不需要把实现倒回缺失态，对已完成代码直接可验。RED 信号适合 coder 自证。两者都进客观门。

### B.3 三 review 合并为一：3 → 1

test 的主体职责（测试是否有效）被 B.2 客观门接管后，三个 review 的剩余职责高度重叠，应合并为**单个 review**：

| 原 review | 剩余职责（客观门接管后） | 去向 |
|---|---|---|
| test review | 测试是否覆盖 **spec 定义的关键行为**（语义覆盖，非有效性） | 并入合并 review |
| spec review | 实现是否逐条满足 spec | 合并 review 主体 |
| code review | 代码质量 / 安全 / 错误处理 | 合并 review 主体 |

**合并不只是省钱，反而更准。** 代码对不对、测试够不够、是否满足 spec——三件事高度耦合。一个 agent 同时握着 spec + diff + tests 才能交叉印证：
- "测试覆盖了 X，但 spec 要的是 Y" ← 分三个 agent 谁都发现不了
- "代码实现了 spec 的 A，但没有任何测试碰它" ← 需要同时看 spec+code+test

分三个 reviewer 的唯一好处是并行省时间，代价是上下文割裂 + 3× dispatch + **无交叉视角**。客观门把最机械的部分（测试有效性）挪走后，单 agent 的剩余负载已显著下降，hold 得住。

**诚实的 trade-off**：单 agent 上下文负载高于单切面 reviewer，可能审得略浅。缓解=把测试有效性这块最重的客观判断移走（已由突变接管），剩下的语义审查负载可控。若某 task 极复杂，leader 可临时升级回多 reviewer——但这是例外，不是默认。

### B.4 合并后的单 task 调用账

```
旧：1 coder + 3 轮 × 3 reviewer + 1 closer = 最坏 11 次
新：1 coder + 3 轮 × 1 reviewer + 1 closer = 最坏  5 次
    且客观门（突变/测试）在 review 前拦截，错误代码根本不触发 reviewer
    → 多数 task：1 coder + 1 reviewer + 1 closer = 3 次
```

最痛的 test 问题用**客观突变信号**根治，不再依赖主观 review；调用次数腰斩；交叉视角反而更强。

### B.5 协议层改动清单

- `RULES.md` 角色表：`op-spec-reviewer` / `op-code-reviewer` / `op-test-reviewer` 三合一为 `op-reviewer`（审 spec 合规 + 代码质量 + 测试语义覆盖）。
- `op_gate.sh` 承接测试有效性：真测试全绿 + 关键函数哨兵突变必红。
- `opplan` 模板：每个关键 step 增加"哨兵突变点"字段（改哪行、怎么改坏、期望测试变红）。
- `RULES.md` review 判定节：客观门（突变 + 测试）不过，review 不必派，直接回 coder fail 轮。
- `op-coder`：RED 阶段失败输出留档进 context.md。

## 落地顺序

```
P0（客观门 = 真测试/build + 哨兵突变）
  ├──→ 根治 test 痛点（假测试被突变戳穿，不再靠主观 review）
  ├──→ 解锁 B.3（test 有效性移走后，三 review 可合并为一）
  ├──→ 解锁 P1（自治有真锚）
  └──→ 缓解 B（错误代码不触发 reviewer 空转）
B.3（三 review 合并）── 依赖 P0，P0 落地后即可做
P2（plan/coder 职责切分）── 独立，可并行推进
P3（状态机裁剪）── 独立，低风险，可随时做
```

**第一步只做一件事**：写 `op_gate.sh`，内含「真测试全绿 + 关键函数哨兵突变必红」，在 `op_close_post.sh` 和每轮 review 前强制调用。

这一步同时拿下三个最痛的点：
1. 质量门槛变真（P0）
2. test 痛点根治（假测试机械暴露，B.2）
3. 解锁 review 合并（B.3，11 次调用 → 多数 3 次）

其余（P1 护栏、P2 职责切分、P3 状态裁剪）皆可后续增量。

---

## 一句话总结

这套系统用**流程协议**替代**判断与验证**。流程能编排，但补不出"代码到底对不对"这个唯一重要的事实。
**把唯一不可伪造的信号（真测试/build 的退出码）设为门槛**，整座状态机城堡才有了立足的地基；否则它编排的只是文件，不是质量。
