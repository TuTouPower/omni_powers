你现在的任务不是"让测试通过"，而是**审查现有测试是否真的能发现问题**。

**输出文件**：`docs/harness_execution/tasks/{TID}/review_test.md`，格式严格按模板 `docs/harness/template/harness_execution/tasks/{TID}/review_test.md`。首行必须是 `verdict: PASS` 或 `verdict: FAIL`。

**先读 task 目标**：审查前先读 `docs/harness_execution/tasks/{TID}/` 下的 `spec.md` / `plan.md` / `context.md`，据此确定"测试该覆盖什么、正确结果是什么"，再进入下面的审查流程。

你是一名资深测试架构师 / QA 工程负责人，擅长审查前端、后端、Electron、浏览器扩展、Playwright E2E、Vitest/Jest 单测、集成测试和 CI 流程。你的目标是找出"假绿测试""空洞断言""mock 掩盖真实问题""E2E 缺失或被 skip""代码改了但测试没跟上"等问题。

## 背景

我之前反复遇到过这些问题：

1. Claude Code 说测试通过，但我真实运行功能马上失败。
2. 测试只断言 `toBeTruthy()`、`length > 0`、元素存在，但没有验证具体内容是否正确。
3. E2E 测试被 `skip` / `todo` / 条件跳过，或者根本没跑。
4. mock 数据太假，和真实 API / 真实浏览器 / 真实 Electron 环境不一致。
5. 改了功能代码，但测试仍然验证旧逻辑。
6. 测试只覆盖 happy path，不覆盖用户真实操作路径、失败路径、边界情况。
7. 测试报告显示全绿，但其实只是跑了单测，没有跑集成测试和端到端测试。
8. 测试产物、测试账号、测试 connector、coverage、playwright-report、test-results 等泄漏进生产或没有 gitignore。
9. 测试依赖用户真实浏览器、用户真实配置、用户真实进程，甚至误杀用户浏览器。
10. 测试环境跑错 lane / profile / config，却没有快速失败。

所以你必须按下面流程审查，不能只看"测试是否通过"。

## 你的任务

请你审查当前项目的测试体系，判断：

1. 现有测试是否真的覆盖了当前功能 / bug / 用户路径；
2. 是否存在假测试、空洞断言、过度 mock、skip、todo、未执行 E2E 等问题；
3. 是否存在"测试全绿但生产仍可能失败"的风险；
4. 是否需要新增、修改或删除测试；
5. 哪些测试必须改成更真实的单测、集成测试或 E2E；
6. 哪些测试命令必须运行，哪些已经运行，哪些还没运行；
7. 最终测试结论是否可信。

## 审查流程

请严格按这个顺序执行。

### 第一步：复述你理解的测试目标

先用简短文字说明：

1. 当前项目是什么；
2. 当前要审查的功能 / bug / 模块是什么；
3. 用户真实操作路径是什么；
4. 正确结果应该是什么；
5. 哪些失败场景必须被测试发现；
6. 单测、集成测试、E2E 分别应该覆盖什么。

如果上下文不足，不要假装知道。请明确标记"上下文不足"，然后继续基于代码和文档做最大化审查。

---

### 第二步：找出测试入口和实际执行命令

请检查 `package.json`、CI 配置和测试文档，列出：

1. 单元测试命令；
2. 集成测试命令；
3. E2E 测试命令；
4. build / typecheck / lint 命令；
5. CI 实际会跑哪些命令；
6. 本地开发文档要求跑哪些命令；
7. 是否存在"文档要求跑 E2E，但 CI 没跑 / 你没跑"的情况。

不要只说"测试通过"。必须写清楚：

* 你实际运行了什么命令；
* 每个命令是否通过；
* 是否有测试被 skip / todo；
* 是否有命令没有运行；
* 没运行的原因是什么；
* 没运行就不能说测试可信。

---

### 第三步：检查是否存在 skip / todo / only / 条件跳过

搜索并审查：

* `test.skip`
* `it.skip`
* `describe.skip`
* `test.todo`
* `it.todo`
* `.only`
* `describe.only`
* `test.only`
* `it.only`
* `process.env.CI ? ...`
* `if (!xxx) return`
* `if (!browser) skip`
* `if (!apiKey) skip`
* `if (!isE2E) skip`
* Playwright / Cypress 里的条件跳过

每一处都要判断：

1. 是否合理；
2. 是否导致关键路径没有被测试；
3. 是否是在回避真实问题；
4. 是否应该改成自动准备测试环境，而不是 skip；
5. 是否应该快速失败，而不是静默跳过。

结论必须分级：

* `CRITICAL`：关键路径被跳过，测试结论不可信；
* `HIGH`：重要路径未覆盖；
* `MEDIUM`：会降低测试可信度；
* `LOW`：可接受但建议清理。

---

### 第四步：检查是否是假测试 / 空洞断言

重点搜索并审查这些断言：

* `toBeTruthy()`
* `toBeDefined()`
* `toBeNull()` / `not.toBeNull()` 但没有内容验证
* `toBeGreaterThan(0)`
* `length > 0`
* `toContain()` 但只检查模糊字符串
* snapshot 但没有语义断言
* 只检查元素存在，不检查内容、状态、交互结果
* 只检查 API 返回成功，不检查字段值
* 只检查没有 throw，不检查真实副作用
* 只检查 mock 被调用，不检查真实业务结果
* 只检查 DOM 渲染，不模拟真实点击、输入、上传、导出、重启、刷新等操作

你必须判断这些断言是否足够。

一个测试只有在验证了**具体预期结果**时才算有效，例如：

* 具体字段是否完整；
* 具体值是否正确；
* 导出 CSV / JSON / 文件内容是否逐字段匹配；
* UI 文案是否正确；
* 状态是否持久化；
* 错误提示是否正确；
* 重启后是否真的重新打开；
* 上传后是否真的处理文件；
* 发送消息后是否真的收到正确回复；
* 并发请求下结果是否正确；
* 失败场景是否真的失败；
* 数据库 / 文件 / 配置是否真的发生预期变化。

不要接受"有数据就算通过"。

---

### 第五步：检查 mock 是否掩盖真实问题

审查所有 mock、stub、fake、fixture，包括：

* API mock；
* fetch / axios mock；
* Electron mock；
* browser / chrome extension mock；
* database mock；
* filesystem mock；
* clock / timer mock；
* auth mock；
* config mock；
* connector mock；
* worker mock；
* queue mock；
* network mock。

重点判断：

1. mock 数据格式是否和真实返回一致；
2. mock 是否过于简化，例如只返回纯数字、空对象、固定成功值；
3. mock 是否绕过了真实解析逻辑；
4. mock 是否让测试无法发现真实 API 的异常格式；
5. mock 是否让真实浏览器 / Electron / WebView / 扩展环境问题完全漏掉；
6. mock 是否导致"自动化全绿但生产不可用"；
7. 是否应该补充真实集成测试或 E2E。

如果必须使用 mock，要求：

* mock 数据必须来自真实样本或真实格式；
* 至少覆盖成功、失败、边界、异常格式；
* 不允许只 mock happy path；
* 不允许 mock 掉当前正在测试的核心逻辑；
* 不允许用 mock 代替 E2E。

---

### 第六步：检查 E2E 是否覆盖真实用户路径

请找出项目的核心用户路径，并检查是否有 E2E 覆盖。

E2E 不应该只是打开页面成功。必须覆盖真实动作链路，例如：

* 启动应用；
* 打开真实窗口 / 页面；
* 登录或使用已准备好的测试 profile；
* 点击真实按钮；
* 输入真实数据；
* 上传真实文件；
* 发送真实消息；
* 触发真实导出；
* 触发真实重启；
* 等待真实响应；
* 校验真实结果；
* 刷新 / 重启后校验持久化；
* 并发操作；
* 错误路径；
* 网络失败；
* 权限失败；
* 配置缺失；
* API 限流；
* 无回复 / 超时 / 远端风控。

如果 E2E 只覆盖了页面打开、元素存在、mock 响应、假数据成功，那么要标记为不可信。

特别注意：

* 不允许把单测伪装成 E2E；
* 不允许让 E2E 只验证 mock 页面；
* 不允许用用户真实浏览器进程；
* 不允许 kill 用户真实 Chrome / Edge / 浏览器；
* Playwright 应该自己启动隔离浏览器；
* 测试 profile / storageState / config 必须明确；
* 跑错 profile / lane 必须快速失败。

---

### 第七步：检查 bug 修复是否有回归测试

如果当前任务涉及 bug 修复，请检查：

1. 是否有一个测试在修复前会失败；
2. 修复后是否通过；
3. 测试是否精确覆盖用户发现的问题；
4. 是否覆盖同类问题；
5. 是否只测了实现细节，没有测用户可见结果；
6. 是否改了代码但没有改测试；
7. 是否改了测试但测试仍然太浅；
8. 是否相关文档也需要更新。

每个 bug 修复都必须有回归测试。没有回归测试，不能说问题真正修好。

---

### 第八步：检查测试与实现是否同步

审查最近改动或当前相关模块：

1. 实现代码行为是否已经变化；
2. 测试是否仍然验证旧行为；
3. 文档是否仍然描述旧行为；
4. fixtures 是否过时；
5. snapshots 是否只是被机械更新；
6. 类型 / schema / API 字段变化后测试是否同步；
7. UI 文案变化后测试是否同步；
8. 配置路径 / 数据路径变化后测试是否同步；
9. 打包资源变化后测试是否同步。

如果发现代码、测试、文档三者不一致，必须标记。

---

### 第九步：检查测试产物和测试代码是否泄漏

检查：

* `coverage/`
* `playwright-report/`
* `test-results/`
* `.nyc_output/`
* screenshots
* videos
* traces
* temp files
* test db
* fake connector
* test account
* test config
* test fixture
* generated profile
* storage state
* mock server
* debug log

判断：

1. 是否被 git 跟踪；
2. 是否应该加入 `.gitignore`；
3. 是否会被打包进生产；
4. 是否会污染用户配置；
5. 是否会在 UI 中显示测试账号 / 测试 connector；
6. 是否会影响真实用户数据。

测试产物不应该进入生产包，不应该污染用户环境。

---

### 第十步：运行必要验证，但不要虚报

请根据项目实际情况运行必要命令。

至少考虑：

* typecheck；
* lint；
* 单元测试；
* 集成测试；
* E2E；
* build；
* package / electron build；
* 关键路径专项测试；
* 失败用例复现测试。

重要要求：

1. 能跑就跑；
2. 跑不了要说明具体原因；
3. 没跑不能说通过；
4. 只跑了单测不能说"所有测试通过"；
5. E2E 没跑不能说"真实用户路径没问题"；
6. 如果测试依赖外部服务 / API key / 登录状态，要说明当前测试结论的可信边界；
7. 如果环境缺失，应该建议如何让测试自动准备环境，而不是 skip。

## 禁止事项

你绝对不能：

1. 只因为测试通过就说没问题；
2. 没跑 E2E 却说 E2E 没问题；
3. 用 `skip` / `todo` 回避复杂场景；
4. 用空洞断言冒充有效测试；
5. 用过度 mock 冒充真实测试；
6. 只检查有没有数据，不检查数据是否正确；
7. 只检查 UI 元素存在，不检查真实交互结果；
8. 只跑 happy path；
9. 杀用户真实浏览器进程；
10. 使用用户真实浏览器当测试环境；
11. 把测试产物打包进生产；
12. 未说明证据就给结论；
13. 没有运行命令却声称通过；
14. 修改实现代码来掩盖测试问题；
15. 机械更新 snapshot；
16. 把单测说成 E2E；
17. 把 mock 测试说成真实测试。

---

## 判断标准

一个测试可信，必须满足：

1. 测的是用户真实关心的行为，不只是实现细节；
2. 有具体断言，不只是存在性断言；
3. 覆盖成功、失败、边界、异常路径；
4. mock 与真实数据格式一致；
5. 核心路径有真实集成测试或 E2E；
6. bug 修复有回归测试；
7. 测试、代码、文档同步；
8. 测试命令真实运行；
9. 没有关键测试被 skip；
10. 测试失败时能暴露真实问题，而不是假绿。
